Mastercard announced Agent Pay on April 29, 2025. Visa announced Intelligent Commerce in April 2025. Both were answering the same problem: the four-party model was built around a human-present checkout, while agentic commerce delegates shopping and payment execution to software.

The model does not need to be replaced. The consumer, merchant, acquirer, network, and issuer still matter. What changes is the trust evidence each participant receives.

## Why the old signals break

The merchant and gateway used to ask whether the session looked human. Agents do not. They may use cloud infrastructure, programmatic browsing, fast request timing, and no cursor or keystroke signals. That can look like the traffic merchants spent years trying to block.

The acquirer and processor used to receive ordinary card-not-present signals. An agent transaction needs additional context: who the agent is, what the consumer authorized, and why this credential is safe to use in this session.

The network used to infer risk from cardholder and merchant patterns. Agentic checkout needs explicit evidence of prior intent, not only behavioral inference.

The issuer used to fall back to live 3DS step-up when risk was ambiguous. That is awkward for delegated commerce. If every useful agent purchase wakes the cardholder in real time, the agent has not actually been delegated authority.

## What the networks added

**Visa's public architecture** combines Visa Intelligent Commerce with TAP. VIC covers tokenized credentials, consumer controls, passkeys, and agent payment flows. TAP covers signed agent requests so merchants and edge networks can distinguish approved agents from malicious bots.

**Mastercard's public architecture** centers Agent Pay on tokenization, authentication, agent registration, and agentic tokens. Mastercard's announcement says Citi and U.S. Bank were helping scale the service in the United States, with cardholder access expected through issuer apps by the 2025 holiday season.

The design difference is real, but the primitives rhyme:

- Registered agent identity.
- Tokenized or agent-bound payment credential.
- Consumer authorization through passkeys, mandates, or issuer controls.
- Evidence that can survive authorization, dispute, and servicing.

## The merchant role changes first

The merchant's problem is not just payment. It is whether to let the agent reach payment at all.

Visa and Cloudflare's TAP work, Mastercard's Agent Pay work, and Web Bot Auth-style verification all move the merchant from human-shape detection to cryptographic verification. A signed request gives the edge or merchant backend a concrete fact: this request was signed by the private key corresponding to a registered agent.

That does not automatically mean "approve the order." It means "evaluate this as an accountable agent, not as an anonymous bot."

---

## The issuer role changes last

The issuer remains the final authorization decision maker. Agentic commerce changes the evidence available to the issuer:

- The credential can be tokenized and scoped.
- The consumer can authenticate at enrollment or instruction time.
- The authorization can point back to a mandate, instruction, or spending policy.
- The agent identity can be available as part of the wider payment context.

That lets the issuer evaluate "does this transaction fit what the cardholder authorized?" instead of only "does this transaction resemble a normal human checkout?"

The public sources do not support hard claims such as all U.S. Mastercard cardholders being enabled by a specific date, or Visa issuers using a specific ISO 8583 value for an agent transaction. Those details should stay out of public architecture writing until the networks publish them.

---

## Where ISO 8583 fits

ISO 8583 remains the authorization message family for card transactions. Agentic commerce can add context around and inside that flow, but public Visa and Mastercard sources do not disclose all network-specific fields and values.

So the safe statement is:

- The payment still routes through existing merchant, acquirer, network, and issuer participants.
- Tokenization and agent metadata can provide additional context for risk and audit.
- Agent intent or mandate evidence can be referenced by the authorization stack.
- Exact field numbers, private data elements, response codes, and issuer rules are network implementation details unless publicly specified.

This distinction matters. A Visa reviewer will forgive an illustrative packet diagram if it is labeled as illustrative. They will not forgive invented field values presented as a published Visa spec.

---

## What is actually the same

Neither Visa nor Mastercard is abandoning the four-party model. The point of both approaches is to make agent transactions compatible with existing acceptance, issuer controls, tokenization, fraud operations, disputes, and settlement.

The rails remain valuable precisely because they already handle consumer protection, merchant acquiring, issuer risk, refunds, chargebacks, and regulated settlement. Agentic commerce adds proof surfaces around them.

## For engineers building in this space

Start with the primitives, not the vendor names.

**Identity:** HTTP signatures, registered keys, timestamp windows, nonces, tags, and key rotation.

**Credential:** token lifecycle, revocation, scope, merchant binding, and agent binding.

**Authorization:** mandate or instruction state, passkey challenge timing, amount drift, partial capture, refunds, and retries.

**Evidence:** request signature, consumer authorization, merchant response, fulfillment state, and dispute files.

The safest integration writing names what is public, labels what is inferred, and avoids pretending private card-network standards are open documentation.
