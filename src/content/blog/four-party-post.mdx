On April 29, 2025, Mastercard announced Agent Pay. On April 30, Visa announced Intelligent Commerce. One day apart, same week, same problem — and they built meaningfully different architectures to solve it. Both announced their Cloudflare partnership on the exact same day in October. That's not a coincidence. That's the standard converging.

This post is about the 4-party model: the settlement architecture that has moved money between consumers, merchants, banks, and networks since the 1970s. It was built around one assumption — a human is present, initiating the transaction, in real time. That assumption no longer holds.

Here is exactly what breaks at each layer when an AI agent initiates a purchase, and how each network rebuilt around it.

## The 4-party model as a layered defense — and why each layer breaks for agents

The 4-party model is not just a settlement architecture. It is a layered fraud prevention system where each participant has a specific responsibility. Understanding which layer breaks — and why — is the key to understanding what both networks had to rebuild.

### Layer 1: Merchant + Gateway — the front line

The first check happens before the payment reaches the banking system. The merchant's website and gateway are responsible for detecting bot behavior at checkout.

Tools: CAPTCHAs, Web Application Firewalls, velocity checks (too many rapid requests from the same IP), device fingerprinting, JavaScript browser challenges.

Goal: Stop automated scripts from brute-forcing card numbers before they generate an authorization request.

**What breaks for AI agents:** Every one of these tools is a false positive machine against legitimate AI shoppers. An AI agent has no mouse movement. It executes at machine speed from a data center IP. Its browser fingerprint is headless Chrome. Its purchasing patterns are algorithmically optimal — which looks exactly like a coordinated card testing attack. A perfectly legitimate agent buying a $90 pair of headphones fails every single front-line check. The merchant blocks the sale before payment even attempts to authorize. If the agent can't pass the front door, nothing downstream matters.

### Layer 2: Acquirer + Processor

If a transaction passes the gateway, the acquirer evaluates it based on merchant history, address verification (AVS), CVV matching, and known malicious IP lists.

Goal: Filter requests that don't match the merchant's normal transaction profile.

**What breaks for AI agents:** The acquirer's risk model is calibrated against human checkout behavior. An agent transaction from an AWS IP with no AVS match triggers every acquirer risk flag. More specifically: the ISO 8583 message has no field that says "this is a legitimate AI agent." Field F022 (POS Entry Mode) has been set to `01` — manual key entry, the ecommerce default since the 1990s — which tells every downstream system to apply human transaction rules. This is the single most important field both networks modified.

### Layer 3: Card Network — the macro view

The network sits in the middle of global transaction flows and has visibility that individual banks cannot have. This is where large-scale ML models run.

Tools: Real-time risk scoring across hundreds of data points, cross-merchant velocity analysis, distributed bot network detection.

Goal: Catch sophisticated, distributed attacks that no single merchant or issuer can see.

**What breaks for AI agents:** The network's ML models are trained on human behavioral baselines. They flag machine-speed transactions from data center IPs as high-risk by design. A legitimate Skyfire agent shopping at Bose.com receives a maximum risk score because it behaves identically to what these models were built to catch. There is also no mechanism to validate that the consumer already authorized this specific intent before the transaction arrived.

### Layer 4: Issuer Bank — the final decision

The issuer receives the network's risk score, combines it with cardholder history, and makes the final authorization decision.

Tools: Internal risk engines, step-up authentication (3D Secure) when signals are ambiguous — an SMS or biometric challenge that proves human presence in real time.

**What breaks for AI agents:** Step-up authentication is the last line of defense against bots. It works by requiring a human to respond in real time. An AI agent cannot respond to a 3DS SMS challenge. If the issuer's risk model triggers step-up — which it will, given the maximum risk score from the network — the transaction fails permanently. There is no fallback. The sale is lost with no recovery path.

<Callout type="insight">
Every layer of the 4-party model's fraud prevention assumes human presence. CAPTCHA assumes a human can solve visual puzzles. AVS assumes a human typed their address. Device fingerprinting assumes a human browser. Behavioral scoring assumes human timing and patterns. Step-up authentication assumes a human can receive and respond to a challenge in real time. AI agents fail every single assumption simultaneously. The entire security stack has to be rebuilt — not patched.
</Callout>

## What each network rebuilt, layer by layer

With that framing, what Visa and Mastercard built becomes precise:

**Layer 1 (merchant/gateway):** Both provide cryptographic agent identity that the WAF can verify — instead of behavioral heuristics that will always false-positive on agents. Visa uses the TAP JWT. Mastercard uses CDN-layer verification by Cloudflare before the request reaches merchant infrastructure.

**Layer 2 (acquirer):** Both add new ISO 8583 fields — a new POS Entry Mode value (Visa `F022=81`, Mastercard equivalent), an agent identifier (`F048`), and an instruction reference (`F126`) — that tell the acquirer's risk model this is a known agent, not a bot.

**Layer 3 (network):** Both add a pre-payment instruction registry. The consumer's pre-authorized intent is stored before the transaction. When the ISO 8583 message arrives, the network validates that the amount, merchant, and agent match what the consumer authorized. Any tamper in transit — decline.

**Layer 4 (issuer):** Both replace step-up authentication with pre-authorized spending policies. The FIDO2 Passkey assertion happens at instruction time, not transaction time. By the time the ISO 8583 message arrives with the new POS Entry Mode value, the step-up has already happened — the issuer loads the agent's spending policy and runs deterministic checks instead of probabilistic scoring.

The layered defense is preserved. Each party still has its responsibility. Every tool has been replaced with one that works for non-human principals.

## How each role in the model changes

### The consumer: from actor to policy author

In the traditional model, the consumer is present at every transaction. They authenticate, enter credentials, approve the payment.

In the AI model, the consumer is present exactly once: at enrollment. They configure a spending policy — category limits, per-transaction cap, daily velocity — and create a FIDO2 Passkey that signs individual payment instructions. After that, the agent operates autonomously within those guardrails until the consumer revokes access.

### The AI agent: new party in the model

There is now a sixth entity that didn't exist before. The agent sits between the consumer and the merchant, interpreting natural language intent and executing the purchase. It holds the consumer's credential (agent-specific token), has a registered cryptographic identity, and signs every request it makes.

### The issuer: two modes

When `F022=81` arrives, the issuer loads the consumer's pre-configured agent spending policy instead of running the standard human behavioral model. The checks are deterministic — balance, per-transaction limit, category, velocity, token status — rather than probabilistic. Citi and US Bank were the first issuers to go live on Mastercard Agent Pay; all US Mastercard cardholders were enabled by November 2025.

### The acquirer: new ISO 8583 fields

For agent transactions: `F002` gets the agent-specific token (not real PAN), `F022=81` signals agent-initiated, `F048` carries the agent identifier, `F126` carries the TAP integrity hash.

### The merchant: where the architectures diverge

**Visa:** TAP SDK embedded in checkout. JWT validation local, under 5ms. Merchant explicitly integrates.

**Mastercard:** Web Bot Auth at CDN layer. No merchant code change for basic verification. Agents use a Dynamic Token Verification Code formatted for standard card fields. Merchants can participate without touching their codebase.

Mastercard optimizes for adoption at the 100M+ merchant scale. Visa optimizes for the richness of the credential the merchant receives.

## Visa Intelligent Commerce: the TAP architecture

<Callout type="tldr">
Visa adds an identity layer before the payment message. The agent authenticates with Visa directly, receives a 90-second JWT, and presents it to the merchant. The JWT carries the payment instruction reference that links the ISO 8583 message to the consumer's original authorization.
</Callout>

```text
Consumer enrolls → VIC enroll_card() → VTS issues VCN bound to agent_id
Consumer creates FIDO2 Passkey on device

--- Transaction ---

Agent → POST tap.visa.com/v1/validate
        (Ed25519 signed, RFC 9421 HTTP Message Signatures)
Visa TAP → validates signature, nonce, timestamp, wallet policy
Visa TAP → issues JWT (TTL 90s, instruction_ref: pi-abc123)
Visa TAP → registers payment instruction in VIC registry

Agent → POST merchant/checkout (Authorization: TAP-1.0 {jwt})
Merchant → validates JWT locally against cached Visa public key

Agent → VIC retrieve_payment_credentials(instruction_ref)
       → returns VCN tokenized_pan

Acquirer → ISO 8583 MTI 0100
           F002: VCN
           F022: 81           ← agent-initiated
           F048: AGNT:agent_id
           F126: pi-abc123:sha256:...  ← TAP integrity hash

VisaNet → validates F126 against VIC registry (amount, merchant)
        → VTS de-tokenizes VCN → routes to issuer

Issuer → loads agent spending policy for token → approves: F039=00

Agent → VIC submit_commerce_signal(instruction_ref, outcome)
```

The key insight: `F126` creates an integrity lock. VisaNet validates that the amount and merchant in the ISO 8583 message match the original payment instruction the consumer authorized. Tampering in transit → decline with response code 58.

**Visa partners:** Cloudflare (WBA) · `tap.visa.com/.well-known/agents/` · `github.com/visa/mcp` · Skyfire · Stripe · Adyen · Worldpay

## Mastercard Agent Pay: the token-first architecture

<Callout type="tldr">
Mastercard extends its existing token infrastructure rather than adding a new identity layer. Agent identity is embedded in the Agentic Token metadata rather than in a separate JWT credential.
</Callout>

```text
Consumer enrolls → Mastercard Token Service issues Agentic Token
                   bound to agent_id + spending policy
Consumer creates Mastercard Payment Passkey (FIDO2)

--- Transaction ---

Agent → sends HTTP request to merchant
CDN layer (Cloudflare) → Web Bot Auth verification
                       → validates agent cryptographic identity
                       → blocks unregistered agents before reaching merchant

Agent → submits Dynamic Token Verification Code (DTVC)
        in existing merchant checkout form
        (no merchant code change required)

Acquirer (Braintree/Checkout.com/Fiserv) → builds ISO 8583
          F002: Agentic Token (not real PAN)
          F022: agent-initiated equivalent
          Agent context in tokenization metadata

Mastercard Network → validates Agentic Token binding
                   → routes to issuer with agent policy context

Issuer → loads agent spending policy → approves
```

The key insight: the token itself carries the agent binding. Trust is embedded in the credential, not in a parallel identity handshake.

**Mastercard partners:** Cloudflare (WBA) · FIDO Alliance · Microsoft · OpenAI · Google · PayPal · Braintree · Checkout.com · Fiserv

## Side by side

<NetworkComparison client:load />

The most important differences are architectural. **Identity verification timing:** Visa TAP verifies before the checkout request reaches the merchant. Mastercard verifies at the CDN layer — earlier, without merchant involvement. **Token design:** Visa's VCN looks like a card number and lives in `F002`. Mastercard's Agentic Token includes agent-binding metadata in the token lifecycle itself. **Merchant integration:** Visa's TAP SDK is explicit. Mastercard's no-code CDN approach requires no merchant code change — which matters enormously at 100M+ merchant scale.

## What's actually the same

Both networks made identical bets on three things.

**Web Bot Auth + Cloudflare.** Both announced their Cloudflare partnerships on October 14, 2025 — the same day. Both use IETF RFC 9421 HTTP Message Signatures as the cryptographic foundation. This is the standard converging in real time.

**FIDO2 Passkeys.** Both require a Passkey-signed consumer instruction before any agent transaction. The consumer's biometric authenticates the specific purchase upfront; the agent executes autonomously within the authorized parameters.

**The 4-party model itself.** Neither network built parallel infrastructure. ISO 8583 still flows from acquirer to network to issuer. Settlement is T+1. Chargebacks exist. The rails are extended, not replaced.

---

## What's still not solved

**Multi-agent authorization.** When a supervisor agent delegates to a subagent, which holds the TAP credential or Agentic Token? Neither network has published a delegation chain protocol.

**Cross-network interoperability.** A consumer with a Visa card using a Mastercard-integrated platform creates ambiguity. TAP key directory and Agent Pay acceptance framework are separate systems — a Visa-registered agent is not automatically recognized by Mastercard's CDN layer.

**Dispute resolution evidence standards.** The exact standard for what constitutes sufficient evidence in an agent-initiated dispute involving a multi-step orchestration workflow has not been formally published.

**PSD2 in Europe.** Strong customer authentication requirements were designed for human authentication. Pre-authorized agent spending policies don't map onto the SCA framework cleanly.

## For engineers building in this space

**Building on Visa:** `developer.visa.com/capabilities/visa-intelligent-commerce` and `github.com/visa/mcp`. The open-source MCP server gives full VIC access through standard Claude/GPT tool calls. Budget extra time for TAP registration and Ed25519 key pair setup.

**Building on Mastercard:** `developer.mastercard.com/agent-pay`. For merchants, the no-code CDN approach is the fastest path. For acquirers, the Agentic Token requestor integration via Mastercard's Secure Card on File service is the integration point.

**Building agents, not infrastructure:** Skyfire KYAPay handles Visa VIC + TAP registration. For Mastercard, the PayPal partnership and Microsoft Copilot Studio integration are the fastest paths to production.

**Building for both networks:** The Web Bot Auth / RFC 9421 layer is shared — the same Ed25519 key pair format works across both. Token issuance and spending policy layers are separate. The cryptographic identity layer converges.
