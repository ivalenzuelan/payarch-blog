There is a strange gap at the center of the agentic payments conversation. The public debate focuses on protocols, but the production problem is a stack problem.

An agentic payment needs at least five layers:

1. Agent identity.
2. Consumer delegation or mandate.
3. Policy enforcement.
4. Payment credential and rail.
5. Reconciliation and dispute evidence.

Visa, Mastercard, Google, Stripe, and x402 all touch the stack, but they do not own the same layer. Treating them as direct substitutes makes the architecture harder to see.

## Layer 1: agent identity

Before an AI agent can initiate a checkout, a merchant needs to know whether the requester is a legitimate agent or hostile automation. That is where Visa TAP, Mastercard Agent Pay, and Cloudflare's Web Bot Auth work begins.

Visa's TAP repository and Cloudflare's technical write-up both point to the same primitive: HTTP Message Signatures. The agent signs a request; the verifier checks the key ID, timestamp, nonce, tag, signature base, and Ed25519 signature. The result is a request that is accepted because it is attributable, not because it imitates human browsing behavior.

This layer does not move money. It answers "who signed this request, and did the signed request change?"

---

## Layer 2: delegation and mandate

Identity is not authorization. A registered agent still needs proof that the consumer authorized the purchase.

Different systems model that proof differently:

- Visa VIC uses tokenized credentials, consumer controls, and passkey-backed authorization concepts.
- Mastercard Agent Pay uses agentic tokens, tokenization, authentication, and agent registration.
- Google AP2 uses signed mandates and verifiable credentials to create evidence from human intent through payment.
- Stripe ACP lets AI platforms and merchants coordinate an application-layer checkout flow, with Stripe preserving ordinary payment-system behavior underneath.

The shared principle is simple: the AI reasoning loop should not hold a broad, reusable payment credential. The credential should be scoped to identity, intent, policy, and context.

---

## Layer 3: policy enforcement

Consumer payments can often be governed by simple limits: merchant, amount, category, velocity, expiration, and revocation. Corporate payments need richer controls: vendor approvals, budgets, purchase orders, cost centers, tax treatment, and ERP reconciliation.

That does not mean all policy belongs in a card network. A practical stack separates responsibilities:

- The wallet, issuer, or network enforces payment credential rules.
- The merchant or PSP enforces checkout and fraud policy.
- The agent platform enforces user intent and orchestration policy.
- Corporate systems enforce company-specific approval and accounting policy.

This is also where prompt injection becomes dangerous. Cryptography can prove that a request was signed; it does not prove that the agent made the right buying decision before signing.

---

## Layer 4: payment rail

For card-based agentic commerce, the rail is still the existing card network stack. Merchants, acquirers, networks, and issuers do not disappear. The change is the evidence that accompanies the authorization: tokenized credential, agent identity, mandate or instruction reference, and policy context.

The public sources do not publish a complete ISO 8583 mapping for agentic payments. So the safest phrasing is that agent context can ride in authorization metadata, private-use fields, tokenization metadata, or network-specific extensions. It is not safe to claim exact values for POS entry mode, private data elements, or response codes without a public network specification.

For x402, the rail is different. The server returns `402 Payment Required`, the client pays, and the client retries with payment proof. That is excellent for API access and agent-to-agent commerce, but it does not automatically recreate cardholder protections, issuer policy, or chargeback workflows.

---

## Layer 5: reconciliation and disputes

Agentic payments create better evidence if the stack is designed correctly:

- The original instruction or mandate.
- The signed agent request.
- The tokenized credential or payment proof.
- The authorization result.
- The order, fulfillment, refund, and support records.

This matters because agentic commerce will create disputes that look different from ordinary ecommerce: "the agent bought the wrong item," "the merchant page manipulated the agent," "a subagent exceeded the orchestrator's budget," or "the user revoked authority before capture." The protocols have to make those timelines reconstructable.

---

## Protocols by layer

**Visa TAP + VIC.** Strongest at card-network agent identity, tokenized credentials, and Visa-controlled payment trust.

**Mastercard Agent Pay.** Strongest at token-first agentic payments built on Mastercard tokenization, authentication, and agent registration.

**Google AP2.** Strongest at portable mandate evidence across payment methods, including cards and stablecoin extensions.

**Stripe ACP / Instant Checkout.** Strongest where the AI platform and merchant checkout surface are integrated and Stripe handles payment orchestration.

**x402.** Strongest for HTTP-native micropayments, API access, and machine-to-machine services.

The likely outcome is not one winner. The likely outcome is layered interoperability: identity at the edge, mandates in the agent ecosystem, tokenization in card networks, and x402 for services where card rails are the wrong shape.

## Open problems

**Multi-agent delegation.** A user may authorize an orchestrator, but the actual checkout could be performed by a subagent. The industry still needs portable evidence for that chain.

**Prompt injection.** Signed mandates protect the final instruction; they do not protect the reasoning path that produced it.

**Cross-network identity.** A Visa-recognized agent, a Mastercard-recognized agent, and an AP2-recognized agent are not automatically the same principal.

**Field-level standardization.** Public documents explain the high-level architecture. They do not yet give builders one universal map for authorization, clearing, disputes, refunds, and issuer servicing.

## tl;dr

The agentic payment stack has five layers: identity, delegation, policy, rail, and evidence. Visa TAP, Mastercard Agent Pay, Google AP2, Stripe ACP, and x402 are not all competing for the same layer. The card networks extend existing rails; AP2 standardizes mandate evidence; ACP integrates AI shopping surfaces with merchants; x402 makes HTTP-native payment practical for APIs.

Payarch is documenting the architecture that connects those layers without pretending private network details are public specs.
