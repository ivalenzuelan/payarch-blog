## What just happened

Four things ran in your browser tab, sequentially:

**1. Key generation.** `crypto.subtle.generateKey` produced a fresh Ed25519 keypair. The 32-byte public key was encoded as URL-safe base64 to form the key ID — the same format Visa's Key Store uses to identify registered agents. The private key is non-exportable and stays in WebCrypto secure memory.

**2. Signature base string.** The simulator constructed the RFC 9421 §2.5 canonical string: a deterministic, ordered serialization of the HTTP request components that will be signed. The CDN reconstructs this exact string from the incoming request and uses it to verify. Changing any covered component — method, path, authority, content-type — produces a different base string and fails verification.

**3. Ed25519 signature.** `crypto.subtle.sign` signed the base string using the private key. The result is always exactly 64 bytes. Ed25519 is deterministic: same key + same input → same signature. But since a fresh keypair is generated on each run, the signature is unique every time.

**4. Signed HTTP request.** The simulator assembled the complete HTTP request with `Signature-Input` and `Signature` headers. This is what an AI agent actually sends to a TAP-enabled merchant. The CDN at the merchant's edge intercepts it before it reaches the origin server.

## Why this matters for AI payments

Traditional payment fraud systems were designed around one assumption: the entity initiating a transaction is a human. Every signal — mouse movement, typing cadence, device fingerprint, IP geolocation — encodes that assumption. An AI agent destroys all of them simultaneously.

TAP replaces behavioral trust with cryptographic trust. It doesn't ask "does this look like a human?" It asks "can this entity prove it is a registered agent, authorized by a known wallet provider, with a valid instruction from a cardholder?"

The HTTP Message Signature (RFC 9421) is the proof mechanism. The keypair is generated once and pre-registered with Visa's Key Store. Every subsequent request is signed with the private key. Verification is stateless, cacheable, and takes less than 1ms at the CDN.

The nonce prevents replay. The timestamp prevents stale requests. The `tag` parameter (`agent-payment-auth` vs `agent-browser-auth`) tells the merchant whether the agent intends to buy or just browse — so the merchant can gate purchase flows separately from catalog access.

## What this is not

The simulator generates real, verifiable signatures. But several things are simulated:

- **The Visa Key Store** — in production, the agent registers its public key with `tap.visa.com` before any transactions. The CDN fetches it from there. The simulator generates a keypair that has never been registered anywhere.
- **The JWT** — Visa issues a signed JWT after successful validation. The simulator shows what the verification steps would confirm, but doesn't issue a real credential.
- **The consumer token** — the `X-Consumer-Token` field in checkout mode is a plausible structure but not a real VTS token.

What is real: the keypair, the RFC 9421 signature base string, the Ed25519 signature, and the signed HTTP request. Run it twice — the key ID, nonce, and signature are different each time.
