This demo shows the cryptographic core of TAP in the clearest possible form: a browser generating an agent key, building a standards-based HTTP signature, and proving that any tampering breaks verification.

No canned signature. No mocked verifier. No server doing the interesting part off-screen. Your tab generates a fresh Ed25519 keypair, constructs the RFC 9421 signature base string, signs it with WebCrypto, and assembles the signed request an agent would send into a TAP-style validation flow. The missing pieces are the production services around it: Visa's key directory, policy checks, replay store, and JWT issuer.

## What the browser just proved

Four concrete things happened in your tab:

**1. Key generation.** `crypto.subtle.generateKey` produced a fresh Ed25519 keypair. In this demo the private key is non-exportable and never leaves WebCrypto. The public key is exported as 32 raw bytes and rendered as a URL-safe base64 fingerprint so you can see that every run produces a distinct agent identity.

**2. Signature base construction.** The simulator assembled the exact string that gets signed: the ordered serialization of the covered request components plus `@signature-params`. Change the method, path, authority, `content-type`, nonce, or tag and you have a different base string.

**3. Ed25519 signing.** `crypto.subtle.sign` signed that base string with the private key. Ed25519 is deterministic for a fixed key and message, so the same input would reproduce the same 64-byte signature. In the demo you get a new signature each run because the keypair and nonce are new each run.

**4. Verification with failure on tamper.** The simulator assembles the signed HTTP request, then verifies the signature using the public key. The tamper test is the payoff: modify a covered field without re-signing and verification fails immediately.

The important thing is not that the browser can produce a pretty demo. It is that the browser can produce a real signature over a real canonical request form, and that signature fails closed when the request is altered.

## Why this matters for AI payments

Traditional fraud systems assume the party initiating a payment is a human. Mouse movement, typing cadence, device fingerprinting, residential IP reputation, funnel timing: all of it encodes that assumption. A legitimate AI agent breaks every one of those signals at once.

TAP changes the question. Instead of asking "does this look human?", it asks "can this entity prove possession of a registered signing key, and is this request fresh, intact, and appropriately scoped?"

In production, the long-lived keypair is generated once and the public key is registered with Visa's directory before traffic begins. Each request is then signed locally by the agent. Signature verification is cheap and cache-friendly; replay protection adds a small stateful nonce check. That combination is what makes TAP workable at the edge.

That distinction is the point. This simulator is valuable not because it impersonates production end to end, but because it isolates the one thing that must be true for the rest of TAP to matter: an agent request can carry cryptographic identity at the HTTP layer, and that identity breaks the moment the request is tampered with.
