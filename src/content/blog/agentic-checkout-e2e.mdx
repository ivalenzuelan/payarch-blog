## The architecture: Visa Intelligent Commerce

Visa Intelligent Commerce (VIC), launched in April 2025, is Visa's complete infrastructure stack for agentic payments. It has five distinct API modules, each solving a different part of the problem.

## Agent-specific tokenization: the credential model

When a consumer onboards their Visa card to an agent platform, three things happen in sequence.

      <div className="text-sm text-slate-600">The issuing bank confirms the cardholder's identity using existing rails (3DS, biometrics, OTP). This is a one-time event per card-agent pairing.</div>
    </div>
  </div>
  <div className="flex gap-4">
    <div className="flex flex-col items-center">
      <div className="w-8 h-8 rounded-full bg-slate-900 text-white font-bold text-sm flex items-center justify-center shrink-0">2</div>

    </div>
    <div className="pb-8 pt-1">
      <div className="font-bold text-slate-900 text-sm mb-1">Agent-bound token issuance</div>
      <div className="text-sm text-slate-600">Visa issues a pass-through network token specific to this agent. It carries an agent binding in the token metadata, cannot be reused by a different agent, and expires if the consumer revokes access.</div>
    </div>
  </div>
  <div className="flex gap-4">
    <div className="flex flex-col items-center">
      <div className="w-8 h-8 rounded-full bg-slate-900 text-white font-bold text-sm flex items-center justify-center shrink-0">3</div>
    </div>
    <div className="pb-2 pt-1">
      <div className="font-bold text-slate-900 text-sm mb-1">Passkey creation</div>
      <div className="text-sm text-slate-600">The consumer's device creates a FIDO2 credential associated with this agent-token pairing. All future payment instructions require a Passkey assertion from this device.</div>
    </div>
  </div>
</div>

```
Token provisioning flow:

consumer → VIC enroll_card({
             card_number,
             agent_id: "skyfire-001",
             controls: { limit_per_txn: 50000, categories: ["retail"] }
           })
         ↓
Visa VTS → step_up_verification(issuer)   // 3DS or biometric
         ↓
issuer   → identity_confirmed: true
         ↓
Visa VTS → issue_agent_token({
             token: "4111xxxxxxxxAGNT",
             binding: { agent_id: "skyfire-001", consumer_did: "did:web:..." },
             controls: { limit: 500, category: "retail", require_passkey: true }
           })
         ↓
consumer → create_passkey(agent_token_ref)  // FIDO2, device-bound
         ↓
device   → fido2_credential_stored
```

At checkout, the agent calls `retrieve_payment_credentials` against VIC with a valid Passkey assertion for the current payment instruction. VIC returns the tokenized credential for this specific transaction. The agent never sees the real PAN. The merchant never sees anything except the tokenized PAN.

<Callout type="insight" title="Why this attack surface is by design">
  Stealing the agent's session gives you a bound token. Stealing the token gives you a credential with no Passkey. Stealing the device gives you a Passkey with no valid payment instruction. Breaking all three simultaneously, in the right sequence, within a 90-second validity window — that is the designed attack surface.
</Callout>

## The Trusted Agent Protocol: cryptographic identity at the HTTP layer

Tokenization solves the credential problem. But before the token is ever used, the merchant needs to know that the incoming HTTP request is from a legitimate agent — not a malicious bot. This is what the Trusted Agent Protocol (TAP) solves.

TAP was launched October 14, 2025, developed by Visa in collaboration with Cloudflare, with initial partners including Microsoft, Nuvei, Shopify, Stripe, Worldpay, and Adyen. The protocol is built on two open standards: **HTTP Message Signatures** (IETF RFC 9421) and **Web Bot Auth** (Cloudflare's IETF draft). Every agent that wants to participate must register with Visa, submit a public key, and appear in Visa's well-known agent directory at `https://tap.visa.com/.well-known/agents/`.

### The signed HTTP request

When a registered agent makes a request to a merchant, it signs the HTTP request using Ed25519 (or RSA-PSS). The `Signature-Input` header specifies which fields are signed:

```http
POST /checkout/complete HTTP/1.1
Host: bose.com
Authorization: TAP-1.0 eyJhbGci...
Signature-Input: tap-sig=("@method" "@target-uri" "@authority"
  "content-digest" "x-tap-agent-id"
  "x-tap-consumer-token" "x-tap-intent");
  keyid="skyfire-agent-001";
  tag="tap-purchase";
  created=1742518234;
  nonce="a3f8c2e1d9b7"
Signature: tap-sig=:MEQCIBx7zK...base64==:
X-Tap-Agent-Id: skyfire-agent-001
X-Tap-Intent: purchase
X-Tap-Consumer-Token: eyJhbGciOiJFUzI1NiJ9...
Content-Digest: sha-256=:RK/0qy18MlB...==:
```

  <div className="flex gap-3 text-sm">
    <code className="shrink-0 font-mono bg-slate-100 px-2 py-0.5 rounded text-slate-700">tag="tap-purchase"</code>
    <span className="text-slate-600">Signals intent to purchase vs. <code>tap-browse</code> for catalog browsing. Merchants can gate purchase flows differently.</span>

### Merchant-side verification

The TAP SDK validates three things in sequence:

```
1. Fetch agent public key:
   GET https://tap.visa.com/.well-known/agents/skyfire-agent-001
   → { public_key: "ed25519:MCowBQYDK2Vd...", status: "active" }
   (cached with 5-minute TTL)

2. Verify HTTP Message Signature:
   reconstruct canonical signature base from Signature-Input
   verify Ed25519 signature against fetched public key
   check: created within 60s ✓, nonce unused ✓
   → valid: true

3. Validate consumer token JWT:
   verify signature against Visa's public key (cached)
   check: exp > now ✓, aud === merchant_id ✓
   → consumer_recognized: true
   → payment_instruction_ref: "pi-abc123"
```

<Callout type="stat" title="Verification latency">
  The entire validation adds <strong>40–80ms</strong> — the time to check the key cache and verify the JWT. At the Cloudflare edge, behavioral intelligence runs in parallel. Akamai also integrated with TAP in December 2025, providing a second independent behavioral signal. <strong>Both must be satisfied</strong> to complete a purchase.
</Callout>

## The ISO 8583 authorization message: what changes

Once the merchant accepts the TAP-verified request and initiates payment, the acquirer (Stripe, Adyen, Fiserv) builds an ISO 8583 MTI 0100 authorization request. This is where the agent context enters the payment rail that has existed since 1987.

Three existing fields are modified, and one private-use field carries the TAP context:

```
Standard human checkout (MTI 0100):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
F002  PAN            4111000000001111      ← real PAN
F003  Processing     000000               ← purchase
F004  Amount         000000008999         ← $89.99
F011  STAN           123456
F022  POS Entry Mode 01                   ← manual keyed
F048  Add'l Data     [empty]
F126  Private Use    [empty]

Agent-initiated checkout (MTI 0100):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
F002  PAN            4111xxxxxxxxAGNT     ← agent-bound token  ★
F003  Processing     000000
F004  Amount         000000008999
F011  STAN           123456
F022  POS Entry Mode 81                   ← agent-initiated    ★
F025  POS Condition  59                   ← agent present
F048  Add'l Data     AGNT:skyfire-001     ← agent identifier   ★
F049  Currency       840                  ← USD
F126  Private Use    pi-abc123:sha256hash ← TAP instruction ref ★
```

<Callout type="insight" title="The most important change">
  <code>F022 = 81</code> is the new POS Entry Mode value that propagates through the <strong>entire authorization chain</strong>. Every downstream system that sees it applies agent-specific risk rules instead of human-behavioral fraud models. The issuer loads the consumer's agent spending policy. VisaNet applies its agent risk ruleset. The acquirer's fraud engine does not flag the machine-speed transaction.
</Callout>

`F126` carries the payment instruction reference from VIC. When VisaNet receives the message, it validates the instruction reference against its payment instruction registry — checking that the amount, merchant, and currency match what the consumer authorized. If they don't match, the authorization fails before it reaches the issuer.

```
Authorization response (MTI 0110):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
F002  PAN            4111xxxxxxxxAGNT     ← echoed
F004  Amount         000000008999         ← echoed
F011  STAN           000123               ← echoed
F038  Auth Code      123456               ← issued by issuer   ★
F039  Response Code  00                   ← approved           ★
F126  Private Use    agent-confirmed      ← TAP echoed         ★
// Total round-trip: ~180ms
```

## The complete VIC API call sequence

For engineers building on VIC directly (rather than through an agent-enabler like Skyfire), here is the full sequence. Visa provides two integration paths: MCP-based (using StreamableHTTP transport) and direct REST API (using X-Pay token authentication and Message Level Encryption).

```typescript
// VIC MCP tool calls — equivalent REST endpoints are also available

// ── ONBOARDING (one-time per consumer-agent pairing) ────────────
const enrollment = await vic.tools.enroll_card({
  user_id: "consumer-123",
  card_number: "4111000000001111",
  expiration_date: "12/27",
  cvv: "123",
  card_type: "credit"
})
// → { token_ref: "vts-AGNT-001", passkey_challenge: "..." }

// Consumer creates FIDO2 Passkey on their device (client-side)
const passkey = await navigator.credentials.create({
  publicKey: parsePasskeyChallenge(enrollment.passkey_challenge)
})
// → credential stored in device secure enclave

// ── AT PURCHASE TIME ────────────────────────────────────────────
// Consumer says "buy this" → agent initiates payment instruction
const instruction = await vic.tools.initiate_purchase_instruction({
  user_id: "consumer-123",
  merchant_id: "bose-com",
  amount: 279.99,
  currency: "USD",
  item_ref: "bose-qc45",
  // Consumer's Passkey assertion proves authorization
  passkey_assertion: await signWithPasskey(passkey, {
    merchant: "bose-com",
    amount: 279.99,
    timestamp: Date.now()
  })
})
// → { instruction_ref: "pi-abc123", status: "authorized", ttl: 300 }

// Agent presents TAP credential at merchant checkout
// (TAP validation happens automatically via HTTP Message Signatures)

// Retrieve tokenized credentials when merchant accepts
const credentials = await vic.tools.retrieve_payment_credentials({
  instruction_ref: "pi-abc123",
  tap_context: tapSignedRequestContext // from TAP handshake
})
// → { tokenized_pan: "4111xxxxAGNT", expiry: "12/27", cvv_token: "..." }

// Credentials are placed in ISO 8583 F002 by the acquirer
// F022=81 is set by the acquirer
// VisaNet and issuer authorize using agent-specific rules

// ── POST-PURCHASE ────────────────────────────────────────────────
// Submit commerce signal — enables dispute resolution
await vic.tools.submit_commerce_signal({
  instruction_ref: "pi-abc123",
  outcome: "success",
  order_id: "bose-order-78234",
  merchant_confirmation: "conf-abc"
})
// This creates the audit trail that makes disputes resolvable
```

<Callout type="tip" title="Don't skip submit_commerce_signal">
  If a consumer later disputes a transaction, VIC needs the full chain: the original payment instruction with its Passkey assertion, the TAP context at checkout, the ISO 8583 authorization record, <em>and</em> the commerce signal confirming fulfillment. The agent is a <strong>witness</strong> to the transaction, not just an executor.
</Callout>

The agent then pays vendor invoices, recurring subscriptions, and utility bills autonomously — capturing the best cashback card per merchant category automatically. <strong>This is the architecture that scales.</strong></p>
  </div>
</div>

## The fraud problem: four layers of defense

The TAP + VIC architecture provides defense in depth. Understanding why this matters requires understanding what the attacks look like.

<div className="not-prose my-6 space-y-3">
  <div className="flex gap-4 items-start">
    <div className="w-8 h-8 bg-slate-900 text-white rounded-lg font-mono font-bold text-sm flex items-center justify-center shrink-0">1</div>
    <div>
      <div className="font-bold text-slate-900 text-sm mb-1">Cryptographic identity</div>
      <div className="text-sm text-slate-600">The HTTP Message Signature is computationally unforgeable without the agent's Ed25519 private key (stored in a secure enclave). The nonce prevents replay. The 60-second validity window limits the attack surface for intercepted requests. An attacker who intercepts a TAP-signed request cannot reuse it 61 seconds later.</div>
    </div>
  </div>
  <div className="flex gap-4 items-start">
    <div className="w-8 h-8 bg-slate-900 text-white rounded-lg font-mono font-bold text-sm flex items-center justify-center shrink-0">2</div>
    <div>
      <div className="font-bold text-slate-900 text-sm mb-1">Cloudflare + Akamai behavioral intelligence</div>
      <div className="text-sm text-slate-600">Even if an attacker obtained a registered agent's private key, Cloudflare's edge intelligence would detect behavioral anomalies — geographic impossibilities, request patterns inconsistent with legitimate shopping, TLS fingerprint mismatches. Akamai provides a second independent behavioral signal. Both must be satisfied to complete a transaction.</div>
    </div>
  </div>
  <div className="flex gap-4 items-start">
    <div className="w-8 h-8 bg-slate-900 text-white rounded-lg font-mono font-bold text-sm flex items-center justify-center shrink-0">3</div>
    <div>
      <div className="font-bold text-slate-900 text-sm mb-1">Agent-bound tokenization</div>
      <div className="text-sm text-slate-600">Even if both identity layers are bypassed, the VCN token can only be activated by a Passkey assertion from the consumer's enrolled device. An attacker needs physical access to the device — and still needs the correct biometric or PIN to generate the Passkey assertion.</div>
    </div>
  </div>
  <div className="flex gap-4 items-start">
    <div className="w-8 h-8 bg-slate-900 text-white rounded-lg font-mono font-bold text-sm flex items-center justify-center shrink-0">4</div>
    <div>
      <div className="font-bold text-slate-900 text-sm mb-1">Payment instruction controls</div>
      <div className="text-sm text-slate-600">The consumer's spending limits and category restrictions are enforced by VisaNet, not the agent. Even a completely compromised agent cannot exceed the consumer's configured guardrails. VisaNet validates the payment instruction reference in F126 — amount tampering in transit is detected and declined.</div>
    </div>
  </div>
</div>

<Callout type="stat">
  Visa's network analyzes over <strong>500 data points</strong> on up to <strong>83,000 transactions per second</strong> and blocked over <strong>$40 billion in fraud</strong> in 2023. The agent-aware extensions don't replace the existing fraud models — they add a new input (<code>F022=81</code> + agent identity) that feeds into the existing scoring system with agent-specific behavioral baselines.
</Callout>

## The protocol landscape: TAP vs x402 vs MCP vs AP2

2025 produced four competing approaches to agentic payments. They are not mutually exclusive — they solve different problems.

### Visa Trusted Agent Protocol (TAP) + VIC

The most complete infrastructure stack for card-network payments. TAP handles merchant-side identity verification. VIC handles the token, authentication, and instruction management. ISO 8583 carries agent context through existing rails.

**Partners:** Microsoft, Shopify, Stripe, Worldpay, Adyen, Cloudflare, Akamai.
**Best for:** Consumer retail purchases, B2B payments on Visa cards, any use case needing real payment credentials.

### x402

Launched May 2025 by Coinbase. By December 2025: 75–100 million transactions, $24 million processed. x402 V2 launched December 2025 with multi-network support and pluggable facilitators.

The mechanism: a client requests a resource → server responds with HTTP 402 + machine-readable payment terms → client pays via stablecoin (USDC) → server delivers resource. No accounts, no subscriptions, no API keys.

```
x402 flow (simplified):
1. Agent: GET /api/market-data
2. Server: 402 Payment Required
   { amount: "0.001", asset: "USDC", network: "eip155:8453", recipient: "0x..." }
3. Agent: GET /api/market-data
   X-Payment: { signed_authorization: "0x...", transaction: "0x..." }
4. Facilitator: verifies payment on Base blockchain
5. Server: 200 OK { data: ... }
```

<Callout type="warning" title="No chargebacks in x402">
  x402 has no network-level reversal. There is no chargeback mechanism — merchants send compensating transfers manually. This makes x402 closer to cash than card payments: ideal for high-frequency API calls, but a liability for consumer purchases that need buyer protection.
</Callout>

**Best for:** API micro-payments, per-call billing, agent-to-agent payments, data licensing, compute payments.

### MCP Payments

Not a competing protocol — it's VIC wrapped in the MCP transport layer. Visa published a remote MCP server at `github.com/visa/mcp` that exposes all five VIC APIs as MCP tools. Any Claude-based agent can trigger a complete VIC payment through a standard MCP tool call with no custom API integration.

### Agentic Commerce Protocol (AP2) / OpenAI + PayPal

OpenAI launched Instant Checkout for Etsy sellers in September 2025 via the Agentic Commerce Protocol, co-developed with Stripe. Google launched Agent Payments Protocol (AP2) around the same time, using x402 for on-chain settlement.

AP2 uses OAuth-based identity rather than public key cryptography — more compatible with existing OAuth ecosystems, but less suitable for headless browser automation where there is no pre-existing account relationship.

### Protocol selection

| Use case | Best protocol |
|---|---|
| Consumer retail, Visa card present | TAP + VIC |
| B2B invoicing, corporate cards | VIC (payment instructions) |
| API micro-payments, per-call billing | x402 |
| Agents built on Claude / ChatGPT | MCP → VIC |
| Agents with existing OAuth accounts | AP2 |
| Cross-network / multi-chain | x402 V2 |

The 2026 bet from most infrastructure teams: TAP + VIC wins for consumer retail. x402 wins for agent-to-agent and API micro-payments. MCP becomes the standard interface. AP2 finds traction where OpenAI's distribution is strongest.

## What is not solved yet

This infrastructure is impressive but early. Four problems remain genuinely open.

<div className="not-prose my-6 grid grid-cols-1 gap-3">
  <div className="border border-amber-200 bg-amber-50 rounded-2xl p-5">
    <div className="font-bold text-slate-900 text-sm mb-2">Multi-agent authorization</div>
    <div className="text-sm text-slate-600">When one supervisor agent orchestrates multiple subagents, who holds the TAP credential? Does each subagent need independent registration? AWS's AgentCore architecture uses a supervisor pattern, but this is not yet standardized in the TAP spec. The IETF working group forming around Web Bot Auth is expected to address this.</div>
  </div>
  <div className="border border-amber-200 bg-amber-50 rounded-2xl p-5">
    <div className="font-bold text-slate-900 text-sm mb-2">Cross-network interoperability</div>
    <div className="text-sm text-slate-600">TAP and Mastercard Agent Pay launched the same week with near-identical architectures but different key directory formats. American Express is adopting Web Bot Auth. The standards bodies (IETF, EMVCo, OpenID Foundation) are engaged, but convergence is 12–18 months away. An agent building for multi-network acceptance today needs to implement multiple credential formats.</div>
  </div>
  <div className="border border-amber-200 bg-amber-50 rounded-2xl p-5">
    <div className="font-bold text-slate-900 text-sm mb-2">Consumer loyalty and intent verification</div>
    <div className="text-sm text-slate-600">Identity and consent can be codified and logged, but an agent's duty of loyalty — whether it is acting for the consumer's interest or the merchant's — is not yet formally defined or measurable. An agent that systematically recommends sponsored products at the consumer's expense is not detectable by the current infrastructure.</div>
  </div>
  <div className="border border-amber-200 bg-amber-50 rounded-2xl p-5">
    <div className="font-bold text-slate-900 text-sm mb-2">Dispute resolution for misunderstood intent</div>
    <div className="text-sm text-slate-600">If a consumer disputes a transaction because the agent misunderstood intent ("I said gym headphones, not studio headphones"), existing chargeback infrastructure was not designed for this. The commerce signals help establish what was authorized, but the liability model for agent interpretation failures is still being worked out by networks and regulators.</div>
  </div>
</div>
