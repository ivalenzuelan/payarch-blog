```text
Provisioning trace:

consumer → VIC enroll_card({
             card_number,
             agent_id: "skyfire-001",
             controls: { limit_per_txn: 50000, categories: ["retail"] }
           })
         ↓
Visa VTS → step_up_verification(issuer)
         ↓
issuer   → identity_confirmed: true
         ↓
Visa VTS → issue_agent_token({
             token: "4111xxxxxxxxAGNT",
             binding: { agent_id: "skyfire-001", consumer_did: "did:web:..." },
             controls: { limit: 500, category: "retail", require_passkey: true }
           })
         ↓
consumer → create_passkey(agent_token_ref)
         ↓
device   → fido2_credential_stored
```

During execution, the agent calls `retrieve_payment_credentials` via VIC with the required Passkey assertion. VIC responds with the transacting token. The agent remains entirely isolated from the unmasked PAN.

<Callout type="insight" title="Credential Attack Surface Constraints">
  Compromising the agent's remote infrastructure exposes only an agent-bound token. Intercepting the token yields a payload lacking the required Passkey assertion. Compromising the physical device requires bypassing the secure enclave to synthesize a valid instruction. The architecture ensures failure domains remain isolated.
</Callout>

## The Trusted Agent Protocol: HTTP Layer Cryptography

Prior to token utilization, merchant infrastructure must deterministicly distinguish registered agents from unauthenticated bot traffic. This requirement is addressed by the Trusted Agent Protocol (TAP).

Developed collaboratively by Visa and Cloudflare, TAP is standardized upon **HTTP Message Signatures** (IETF RFC 9421) and **Web Bot Auth**. Participating agents persist their public keys in the Visa registration directory (`https://tap.visa.com/.well-known/agents/`).

### Protocol Construction

A registered agent computes an Ed25519 (or RSA-PSS) signature covering specific request attributes. Below is an example of the resulting payload:

```http

Signature-Input: tap-sig=("@method" "@target-uri" "@authority"
  "content-digest" "x-tap-agent-id"
  "x-tap-consumer-token" "x-tap-intent");
  keyid="skyfire-agent-001";
  tag="tap-purchase";
  created=1742518234;
  nonce="a3f8c2e1d9b7"
Signature: tap-sig=:MEQCIBx7zK...base64==:
X-Tap-Agent-Id: skyfire-agent-001
X-Tap-Intent: purchase
X-Tap-Consumer-Token: eyJhbGciOiJFUzI1NiJ9...
Content-Digest: sha-256=:RK/0qy18MlB...==:
```

### Edge Verification

The merchant's edge infrastructure or application server evaluates the request systematically:

```text
1. Key Material Resolution:
   GET https://tap.visa.com/.well-known/agents/skyfire-agent-001
   → { public_key: "ed25519:MCowBQYDK2Vd...", status: "active" }

2. Signature Validation:
   - Compute the required canonical signature base.
   - Validate Ed25519 signature against the registry.
   - Verify timestamp bounds and nonce novelty.
   → Result: valid

3. Token Authorization:
   - Validate JWT cryptographic signature.
   - Ensure the audience claim ('aud') matches the merchant origin.
   → Result: consumer_recognized
```

<Callout type="stat" title="Execution Overhead">
  Complete verification introduces approximately <strong>40–80 milliseconds</strong> in latency. Furthermore, network edges (e.g., Cloudflare and Akamai) process proprietary behavioral intelligence concurrently. Both cryptographic validation and behavioral metrics must coalesce to authorize the request.
</Callout>

<Callout type="insight" title="Systemic Impact of F022 Modifications">
  Deploying <code>F022 = 81</code> re-routes the transaction through an alternate logic flow across the entire settlement chain. VisaNet and issuer endpoints evaluate the request utilizing agent-oriented parameters, disregarding human-interaction heuristics completely.
</Callout>

Element `F126` enforces transaction integrity. VisaNet structurally validates the `F126` instruction reference against the initial payload verified by the consumer's Passkey; deviations in currency, amount, or merchant identity guarantee an immediate decline.

## API Execution Sequence

For architects implementing deep integration rather than delegating to enabling platforms, the complete execution requires synchronizing the five modules. Visa provides bindings via the Model Context Protocol (StreamableHTTP) and via native REST interfaces.

```typescript
// Stage 1: Enrollment (Asynchronous preprocessing)
const enrollment = await vic.tools.enroll_card({
  user_id: "consumer-123",
  card_number: "4111000000001111",
  expiration_date: "12/27",
  cvv: "123"
});
// → { token_ref: "vts-AGNT-001", passkey_challenge: "..." }

// Client-side execution bridging FIDO2 specifications
const passkey = await navigator.credentials.create({
  publicKey: parsePasskeyChallenge(enrollment.passkey_challenge)
});

// Stage 2: Intent Origination (At transaction juncture)
// Consumer triggers agent; agent formulates explicit instruction parameters
const instruction = await vic.tools.initiate_purchase_instruction({
  user_id: "consumer-123",
  merchant_id: "bose-com",
  amount: 279.99,
  currency: "USD",
  item_ref: "bose-qc45",
  passkey_assertion: await signWithPasskey(passkey, {
    merchant: "bose-com",
    amount: 279.99,
    timestamp: Date.now()
  })
});
// → { instruction_ref: "pi-abc123", status: "authorized" }

// Stage 3: TAP Exchange & Credential Injection
// Assuming TAP handshake succeeds via HTTP signatures
const credentials = await vic.tools.retrieve_payment_credentials({
  instruction_ref: "pi-abc123",
  tap_context: tapSignedRequestContext
});

// Stage 4: Resolution & Telemetry
// Required to finalize audit and enable programmatic reversals
await vic.tools.submit_commerce_signal({
  instruction_ref: "pi-abc123",
  outcome: "success",
  order_id: "bose-order-78234",
  merchant_confirmation: "conf-abc"
});
```

<Callout type="tip" title="Commerce Signal Integration">
  Bypassing the <code>submit_commerce_signal</code> endpoint corrupts the dispute validation chain. Financial institutions require the initial intent parameters, the TAP signature, the ISO 8583 response, and the agent's signal telemetry to adjudicate liability correctly.
</Callout>

## The Protocol Ecosystem: Analyzing TAP, x402, and AP2

Current engineering efforts have produced multiple frameworks designed for intersecting use cases:

### Visa Trusted Agent Protocol (TAP) + VIC
Employs HTTP signatures combined with standard Visa network tokenization. Optimized for large-scale retail commerce, executing on legacy rails via ISO 8583. Supported by major processors (Stripe, Adyen, Worldpay).

### x402 (Coinbase)
Facilitates programmatic compensation via HTTP 402 ("Payment Required") responses resolving to smart contracts. Settles deterministically via stablecoins (USDC/EURC). Contains zero inherent reversal or chargeback mechanisms. Highly performant for API orchestration and sub-dollar machine-to-machine licensing.

### Agentic Commerce Protocol (AP2)
Supported primarily by OpenAI and Google. Leverages established OAuth scopes rather than bespoke cryptographic payload definitions. Less demanding computationally, though susceptible to friction during initialization within headless environments where standard single-sign-on (SSO) contexts do not persist.

| Architectural Need | Optimal Specification |
|---|---|
| Large-basket commerce, chargeback dependencies | TAP + VIC |
| Corporate API utilization, high-volume transactions | x402 |
| Corporate card invoicing, strict velocity limits | VIC (Instruction Layer) |
| Environments requiring standard OAuth implementations | AP2 |
