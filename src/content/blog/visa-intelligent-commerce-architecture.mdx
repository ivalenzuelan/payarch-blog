Visa Intelligent Commerce is easiest to misunderstand if you start at the API docs. The API is only one piece. A production payment crosses a credential registry, a passkey-backed instruction store, an HTTP signature verifier at the edge, VisaNet authorization logic, and a fee program for enriched transaction data.

The object to track is the authorization packet. By the time it reaches the issuer, three VIC systems have left marks on it: the VIC API produced the credential and instruction reference, TAP verified the agent request, and VDCAP supplied the data that makes the transaction cheaper and easier to score.

## The System Map

VIC API owns the payment lifecycle. TAP owns request identity. VDCAP owns the data incentive. Treating them as one product is how integrations become blurry.

## System 1: VIC API

The VIC API is a lifecycle, not a checkout endpoint. The split matters because each operation establishes a different control point.

```text
enroll_card                     -> token_ref + passkey challenge
initiate_purchase_instruction   -> instruction_ref
update_or_cancel_instruction    -> revoke or adjust before checkout
retrieve_payment_credentials    -> VCN + expiry + enriched data
submit_commerce_signal          -> outcome, order_id, fulfillment state
```

Enrollment happens once per consumer-agent pairing. Visa verifies the cardholder with the issuer, then issues an agent-bound Virtual Card Number. The real PAN stays in the token vault. The agent, merchant, and acquirer see the VCN.

Instruction initiation is the point of human authorization. The consumer's device signs a FIDO2 Passkey assertion for a specific merchant, amount, and purchase context. The response is an `instruction_ref`, which becomes the transaction's audit anchor.

Credential retrieval happens after the agent reaches checkout and TAP verification has passed. VIC checks the instruction state, agent identity, and TAP context before releasing the VCN. A cancelled instruction returns no credential.

The commerce signal closes the loop. `instruction_ref`, authorization code, and `order_id` become the evidence chain for disputes and fraud review.

## System 2: TAP

VIC API and TAP run at different layers. VIC asks whether this consumer has an active purchase instruction for this amount at this merchant. TAP asks whether the HTTP request came from a registered agent, and whether anything changed after the agent signed it.

That second question has to be answered before VIC is called, which is why TAP runs at Cloudflare's edge.

### The key infrastructure

Every agent platform that integrates VIC registers an Ed25519 keypair with Visa. Public keys are served from `tap.visa.com/.well-known/agents/{agent_id}`, following the JWKS pattern, and Cloudflare edge nodes cache them for 5 minutes. Private keys stay inside the agent platform's secure enclave.

### The signed request

Every checkout request from a TAP-registered agent carries an RFC 9421 HTTP Message Signature:

```http
POST /checkout/complete HTTP/1.1
Host: bose.com
Content-Type: application/json
Content-Digest: sha-256=:base64hash==:
Signature-Input: tap-sig=("@method" "@authority" "@path"
  "content-type" "content-digest" "x-tap-agent-id" "x-tap-nonce");
  keyid="skyfire-agent-001"; tag="tap-purchase";
  created=1742518234; nonce="a3f8c2e1d9b7"
Signature: tap-sig=:MEQCIBx7zK...base64==:
X-Tap-Agent-Id: skyfire-agent-001
X-Tap-Nonce: a3f8c2e1d9b7
X-Tap-Consumer-Token: eyJhbGciOiJFZERTQSJ9...
```

The `Signature-Input` header defines the signing base: the exact headers and request components covered by the signature. Change any covered field after signing and verification fails. The `content-digest` covers the body, so cart and amount tampering also fails verification.

Four fields in this header carry security weight:

- **`tag="tap-purchase"`** separates payment intent from catalog browsing. Merchants gate checkout on this value specifically, not on TAP signatures generally.
- **`nonce`** is single-use. Visa stores nonces after first sight; replays are rejected at stage 4.
- **`created`** gives Cloudflare the replay window. Requests older than 300 seconds are rejected at the CDN layer.
- **`X-Tap-Consumer-Token`** is a 90-second JWT issued by the VIC API at instruction time. It binds "who is this agent" from TAP to "what is this agent allowed to buy" from VIC.

Total budget: roughly 2-4ms at edge, plus 1-3ms in the merchant SDK if the merchant verifies locally. Under 10ms either way.

### Merchant-side integration

Cloudflare-fronted merchants receive `X-Tap-Verified: true` and `X-Tap-Agent-Id` on requests that cleared the pipeline. Most merchants pass those headers into fraud scoring and checkout policy.

Merchants outside the Cloudflare path, or merchants that want an audit-independent check, run Visa's TAP SDK and replicate the same checks at origin.

## System 3: VDCAP

The Visa Digital Commerce Authentication Program is the economic layer. It does not decide whether the transaction is approved. It changes the merchant's cost when the authorization carries cleaner data.

VDCAP rewards two levels of enrichment:

- **Tier 1: enriched data.** Device ID, IP address, email hash, and AVS data qualify the merchant for a 0.05% interchange reduction.
- **Tier 2: enriched data plus network token.** The same data, paired with a Visa Network Token, qualifies for a 0.10% reduction.

The architecture matters because the agent stack already has this data. The agent platform knows the device and IP. VIC has the consumer and AVS context. The merchant does not have to invent a new data collection process; it passes through what the credential flow already produced.

```text
device_id      -> F048 Tag 91
ip_address     -> F048 Tag 33
email_hash     -> F048 Tag 42
network token  -> F002 / F035
avs_data       -> issuer risk inputs
```

On a $10M monthly card volume, 0.10% is $10,000 per month. Engineering teams integrate VIC for identity and fraud control. VDCAP is the finance argument that helps the project get approved.

## Full Transaction Flow

<VicTransactionFlow client:load />

The important boundary is credential possession. The agent can decide what to buy, but it does not hold the consumer's real card. It receives a signed request path, an instruction reference, and a tokenized payment credential only after the VIC API and TAP checks agree.

The second boundary is message compatibility. The merchant still sends an ISO 8583 authorization request through the acquirer. Visa did not create a parallel rail for agent payments; it gave the existing rail enough new context to stop treating the agent like a bot.

## ISO 8583 Packet Anatomy

<Iso8583PacketDiagram client:load />

The field to watch is `F022`. Traditional ecommerce commonly travels as card-not-present traffic with human checkout assumptions. `F022=81` tells downstream systems that the transaction was agent-initiated. That lets the issuer evaluate the registered instruction and token policy instead of forcing a real-time 3DS challenge.

`F002` carries the agent-bound VCN, not the consumer's real PAN. `F048` carries agent identity and VDCAP data. `F126` carries the integrity link back to the purchase instruction. If amount or merchant has been changed after instruction registration, VisaNet can fail the authorization before it reaches a normal issuer approval path.

Backward compatibility is the design constraint. The routing is still VisaNet. Settlement is still the normal card-network cycle. The new agent semantics ride in fields that already exist.

## Production status

As of April 2026:

- **Sandbox**: available globally across all five Visa regions (North America, Asia-Pacific, Europe, CEMEA, Latin America & Caribbean)
- **Production**: gated by partner approval; contact Visa for access
- **VDCAP**: launched US and Canada, April 2026
- **TAP open source**: published October 14, 2025 at `github.com/visa/trusted-agent-protocol`
- **VIC MCP server**: `github.com/visa/mcp`; MCP-compatible agents can call the VIC operations as tools

Partners confirmed in production or advanced sandbox: Cloudflare, Skyfire, Stripe, Adyen, Worldpay, Crossmint (virtual card issuance via Visa Intelligent Platform), Ramp (B2B deployment, April 1, 2026).

## Integration paths

There are three integration paths. The choice is mostly about control.

**Direct Visa integration.** Register the agent identity, implement the five VIC operations, and sign checkout requests according to RFC 9421. Best for agent platforms that will serve many merchants and need direct control over the credential lifecycle.

**VIC via MCP server.** `github.com/visa/mcp` exposes enrollment, instruction initiation, and credential retrieval as MCP tools. Best for agents already built around MCP-compatible runtimes.

**Via Skyfire or Crossmint.** These products sit above VIC and handle TAP registration, enrollment, and credential operations. Best for consumer-facing apps that want a payment stack without a direct Visa partnership.

---

## What VIC does not solve

**Multi-agent authorization.** The current VIC spec assumes one agent per payment instruction. When a supervisor agent delegates to a subagent, and the subagent initiates the checkout, which agent ID is in the TAP credential? The `instruction_ref` was registered against the supervisor's session. Neither VIC nor TAP specifies how delegation chains propagate through sub-agents.

**Cross-network interoperability.** A consumer with a Visa card on a Mastercard-integrated platform creates a gap. Mastercard's verification layer does not recognize TAP credentials, and VIC does not issue Mastercard Agentic Tokens. A multi-network agent has to implement both.

**Cross-border payments.** VDCAP and the VIC API apply to card-network authorizations. International wire transfers run on SWIFT `pain.001` / `pacs.008` messages, where none of this architecture applies. The SWIFT Payment Initiation Relay Rulebook (March 2025) introduced a Forwarding Agent role for automated systems, but it has no agent identity or instruction registry concepts.

**Prompt injection defense.** TAP protects the checkout request after the purchase instruction exists. It does not protect the agent's decision-making before the cart is created. If a merchant page manipulates the agent's shopping logic, VIC has no visibility into that reasoning path.
